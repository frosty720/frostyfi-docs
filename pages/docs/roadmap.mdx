# Roadmap

> Gate-driven, not date-driven. The token launches from strength — when the product is proven — not on a calendar.

import { Callout } from 'nextra/components'

## Current status

**🚀 MVP live** at https://app.frostylabs.ai — visual workflow builder, AI chat, multi-chain wallet operations (1700+ chains via ThirdWeb), MCP integration, A2A / ERC-8004 agents, x402 micropayments, and the Frosty Engine for scheduled and triggered workflows.

## Why gates, not dates

A fair launch with no presale raises **no capital** — so there is zero reason to rush the token, and every reason to launch it with live utility and real traction. Each phase below produces the preconditions the next one needs. The token launches only when the [launch gates](#launch-gates) are met.

## Phase 1 — SaaS foundation

Goal: a self-funding product. Highlights:

- **One AI provider, one balance** — consolidate on OpenRouter for text, image, and video, in both chat and workflows.
- **Per-user spending caps** — each user gets a hard usage cap enforced at the provider level, so no single user can run up the bill. Free tier runs on low-cost models with a small cap; nothing is unlimited.
- **USDC subscriptions** — migrate from NFT subscriptions to a direct USDC subscription contract on Base, with a card fallback for non-wallet users.
- **Affiliate program** — referral attribution plus an earnings & links dashboard.
- **Deploy any workflow as an agent** — turn a workflow into an x402-monetized, ERC-8004-registered A2A agent from the builder.
- **Polished agent creation** — wire the deploy step directly into the create flow.
- **Showcase:** workflows + x402 + ERC-8004.

## Phase 2 — Growth & the native model

Goal: manufacture the token's preconditions.

- **Expand use cases** to grow subscribers past the self-funding line — starting with a multi-chain liquidation-prevention agent, plus x402 data nodes and verified-execution workflow templates.
- **Train & validate FrostyG4** — FrostyFi's own fine-tuned model (Gemma-based), starting small (E4B) and graduating to 26B. Until it beats the best stock model, we simply route to stock models — no premature infrastructure.
- **Audit the token contracts** — finalize and third-party audit `frosty-core` so it's launch-ready.
- **Publish the full docs** — tokenomics, POL, funding, and this roadmap, ready for launch.
- **Self-hosting** (Docker/Coolify) — a self-host option for teams who want to run FrostyFi on their own infra. Planned for a later release; the product is **SaaS-only today**.

## Phase 3 — Token & POL launch

Goal: launch Permafrost from strength.

- **Permafrost ($PERMA) fair launch** + PERMA/USDC liquidity on Base. No VC, no insiders. See [Tokenomics](/docs/tokenomics).
- **Subscribe → buy PERMA → POL** — subscription revenue continuously buys PERMA into DAO-owned liquidity. See [POL & Funding](/docs/pol).
- **PERMA-as-payment** for the native FrostyG4 model — real utility, on day one.
- **POL + DAO governance** via soulbound POLAR.

## Launch gates

The token launches **only when all four are met**:

1. **Traction & revenue** — a healthy base of paying subscribers and several consecutive months of operating surplus.
2. **Native model** — FrostyG4 demonstrably outperforms the best stock model on FrostyFi's workflow evaluations and is live in production.
3. **Contracts** — `frosty-core` fixed and third-party audited, with the buyback router, liquidity, and POL flows tested end-to-end.
4. **Docs live** — complete public tokenomics, POL, funding, and roadmap documentation published.

<Callout type="info">
We will publish progress against these gates as they are met, so the community can see exactly where things stand before launch.
</Callout>

## The promise

> No team allocation. No VC. No presale. No insiders. The founder buys PERMA on the open market, like you. The developer is paid from revenue to keep building — not from a token bag to sell.
